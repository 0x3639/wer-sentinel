# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

A **Docusaurus documentation site** whose purpose is to help humans and AI agents
**implement a Sentinel service node that runs from inside the Syrius wallet**. It combines
provenance-tagged background on Zenon Sentinels, a code-verified investigation, a design
spec, and a build roadmap. The Syrius source is vendored as a git submodule at `syrius/`
(pinned `v0.1.0-alphanet-8`); all `file:line` references point into it.

## Layout

```
docs/
  intro.md                              Landing / overview (served at site root, slug /)
  understanding/                        What a Sentinel is + provenance map
  evidence/investigation.md             Code-verified feasibility analysis (+ raw Q&A appendix)
  specification/sentinel-service-layer.md   The design contract (v1)
  implementation/roadmap.md             Ordered build tasks with code anchors
  implementation/agent-guide.md         How an AI agent should use these docs
  community/forum-post.md               Community pitch

docusaurus.config.js, sidebars.js, src/css/  Site config + theme
syrius/                                 Syrius source (git submodule) — DO NOT edit here
```

## Commands

```bash
npm install            # install site deps
npm start              # dev server (http://localhost:3000)
npm run build          # static build into ./build  (run this to verify changes compile)
npm run serve          # preview the production build
```

`git submodule update --init --recursive` if `syrius/` is empty after clone.

## Conventions that govern doc edits

1. **Provenance tags must stay accurate.** `[DOC]` = documented in a cited primary source ·
   `[INF]` = inferred (e.g. from code) · `[OPEN]` = unresolved. Never promote `[INF]`/`[OPEN]`
   to `[DOC]` without a real citation. The "feeless middle-layer" reading is always `[INF]`;
   specific Sentinel collateral figures are always `[OPEN]`.
2. **Hold the conflict.** Zenon's whitepaper / lightpaper / greenpaper describe Sentinels in
   three conflicting ways. Present all three, attributed; don't collapse them. (The site
   *adopts* the greenpaper "service edge" reading for the build, but flags it as a choice.)
3. **Code is ground truth.** Any claim about Syrius needs a `file:line` receipt into
   `syrius/`. When the submodule is bumped, re-verify references before trusting them.
4. **Markdown is CommonMark.** `markdown.format: 'detect'` in the config means `.md` files
   are *not* MDX — inline `<` and `{` in prose are safe. Use `.mdx` only if you deliberately
   want JSX. Keep code-like content in fences.

## Docs structure rules (Docusaurus)

- Each doc needs frontmatter; nav order comes from `sidebar_position` + each folder's
  `_category_.json`. The sidebar is autogenerated from `docs/`.
- Don't add a second `# H1` if frontmatter sets `title:` (avoids duplicate titles).
- `onBrokenLinks` is `warn`, so a broken cross-link won't fail the build — check the build
  log for warnings.

## External primary sources

The Zenon papers cited in `docs/understanding/sources.md` live in
[**zenon-developer-commons**](https://github.com/TminusZ/zenon-developer-commons) as PDFs at
the repo root (URL-encode `(`→`%28`, `)`→`%29`, `:`→`%3A` when linking). Line numbers in
that file refer to old local Markdown conversions and do **not** map to the PDFs — relocate
by section context.

## Deploying

Push to `main` → `.github/workflows/deploy.yml` builds and publishes to GitHub Pages. Update
`url` / `baseUrl` / `organizationName` / `projectName` in `docusaurus.config.js` to match the
real repo before the first deploy (see README).
